Part 2 В· Three Principles

Principle 1 В· Structure beats memory

You cannot rely on people remembering how a project works.

  • The PI is busy
  • The original postdoc has left
  • Even you will forget in six months

. . .

A clear directory structure makes a project understandable even without the author.

The structure itself communicates:

  • Where things live
  • What belongs where
  • What the project is doing

Compare:

A repo where you have to ask someone what to run.

vs.

A repo where you open it and immediately see README.md, 03_data/, 04_data_analysis/, 05_figures/.

. . .

One of these is a gift to future collaborators.

Principle 2 В· One project вү  one person

Research repositories are often built as if only one person will ever use them.

  • No README
  • No explanation of how to run the analysis
  • Folder names that only make sense to the author

. . .

Repositories should be designed for collaboration and handover, not individual workflows.

Even if you work alone today, you will eventually:

  • Have a new student join
  • Collaborate with someone from outside your group
  • Hand the project to someone else

Ask yourself:

Could someone who doesnвҖҷt know me get started in this repository with no help from me?

. . .

If the answer is no вҖ” thatвҖҷs the problem to fix.

Principle 3 В· Onboarding is part of reproducibility

Reproducibility is often framed as a technical problem:

  • Use version control
  • Record your environment
  • Write clean code

. . .

If a new researcher cannot tell how to start, the project is not truly reproducible.

Onboarding documentation is part of reproducibility, not optional extra work.

Onboarding documentation answers:

  • What is this project?
  • How do I run it?
  • Where do I put my own work?
  • Who do I ask if I have a question?
  • What are the conventions we use here?

A 30-minute investment in writing this saves days of confusion later.

README.md вҖ” the front door

The README is the first file anyone reads.

It should answer:

  • Who maintains it?
  • How do I get started?
  • Where do I find things?

A README does not need to be long.

Four sections. Plain English. No jargon.

The onboarding document is for new people joining the lab.

It should explain:

  • How we use this repository
  • Conventions we follow (naming files, committing, reviewing)
  • Where to ask questions

Write it as if youвҖҷre talking to someone on their first day.

  • How to get set up
  • Repository conventions

Key conventions:

  • Keep the GIN-Tonic top-level numbering unchanged
  • 03_data/001_rawdata/ is read-only вҖ” never overwrite source data
  • Analysis code lives in 04_data_analysis/
  • Use the same numbering for analysis steps: 001_cleaning.R, 002_model.R
  • Generated outputs go in 05_figures/ or 03_data/990_processed_data/
  • Environment files can live in 07_miscellaneous/

Copy this folder for every new project. Give it a meaningful name.

вҖңIt worked on my machineвҖқ is one of the most common reproducibility failures.

Best practices that prevent chaos

Naming and file conventions

  • Number analysis steps clearly: 001_cleaning.R, 002_qc.R, 003_model.R
  • Use one naming style consistently: lowercase with underscores is usually safest
  • Use descriptive names, not final_v2_really_final.R
  • Avoid spaces and special characters in file names
  • Keep one idea per file name: manhattan_plot.png, not plot_new_latest.png
  • Keep dates in YYYY-MM-DD format when you need dated outputs

Documentation and tracking

  • Add a short README.md to major directories
  • Comment why a step exists, not only what the code does
  • Record important thresholds, filters, and random seeds
  • Keep a simple analysis catalogue with: analysis ID, purpose, input data, main script, status, and output location

Small conventions reduce repeated confusion later.

Best practices that prevent chaos, continued

Data and output rules

  • Never overwrite raw data
  • Keep generated data separate from source data
  • Avoid overwriting important outputs
  • Create versioned output files when needed
  • Use relative paths so the project still works when moved

Quick checks

  • Can a new person tell what each script is for?
  • Can they see which file runs first?
  • Can they tell which files are source data and which are generated?
  • Can they rerun the project on another machine?

If not, the structure still needs work.
